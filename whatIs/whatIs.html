Overview

ConfigHub is an Application Control Platform (ACP). It is capable of centralizing configuration for the entire development stack, via private, secure, versioned configuration repositories. In contrast to infrastructure-centric tools (i.e. Chef, Puppet, Salt, or DIY) ConfigHub is unique because our focus is on managing large and diverse configuration sets for all code developed and maintained by the enterprise.

The unique approach of managing configuration via context definitions, allows abstraction of each application segment in a way that completely eliminates all configuration duplication and thus enables simple, safe and fast configuration management.

ConfigHub platform is free, it is downloaded and installed locally.

What is context?

At the heart of ConfigHub is context. It eliminates configuration duplication, greatly simplifies configuration in large systems, and lets you define your application space in a way not limited by any pre-determined conditions.

The traditional configuration property is defined as a key-value pair: 

key: value
The limitation of this definition is that it does nothing to describe who should know about this property, and therefore it will get duplicated for each consumer (application/instance).

Context properties and files

To overcome this limitation, we expanded the definition of the property to add context to the value. A context property looks like this: 

key: { context: value }
Now, context describes a subset of applications that should know about this property.

Furthermore, a key may have multiple values because the context for each value has to be unique within the key. And the same logic applies to files. Currently a file name has to be unique for a specific path. However, with addition of context, multiple files with the same name can exist in the same path. 

key: {
    context(1): value,
    context(2): value,
    ...
    context(n): value
}
path/filename: {
    context(1): file-content,
    context(2): file-content,
    ...
    context(n): file-content
}

Repository context

A repository, when created, defines a context that will be used by context properties and files. Repository context is defined to best describe your application space. It may be as simple as 

Environment > Application
or as "wide scoped" as you need it to be - i.e. 
Company > System > Platform > Environment > Application > Instance

How applications receive configuration

Applications receive configuration via PULL request from ConfigHub platform just by specifying their running context. Application's context has to be fully-specified, meaning, each element of the context has to be specified. With fully specified context, only a single, most specific context property value and context file is returned.

ConfigHub platform uses RESTful Json API for both configuration PUSH and PULL. There are also several language specific APIs available on the download page.

Setup and fault-tolerance

In basic setup, ConfigHub is running either as a Docker container or on metal/cloud installation. It requires connection to two databases. The "Main" database is used to store all repository and user data, and it is recommended for this database to have a replica. The second database is an archive of all API request, and replication is optional (depending on how relevant this information is to you.)

In this setup, applications will use either a language specific API, or a RESTful JSON API to pull configuration directly from the ConfigHub app server.

DNS managed failover setup

Having a slave database attached to a hot-standby ConfigHub instance allows for a fast failover with minimal downtime. With an update to the DNS record and a switch of the database from Slave to Master setup, full operational state is recovered.

Config caching via proxy agent

Local cache agent serves as a proxy to ConfigHub. Application request configuration locally from the agent, which will then fetch configuration from ConfigHub service and store it locally as last-know-configuration for each requested context.

If, for whatever reason, configuration cannot be pulled from ConfigHub, last known configuration can be retrieved from the local agent. Since all per context locally cached configuration is kept as files, they can be modified manually if need be.
